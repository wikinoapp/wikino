## 概要

<!--

WikinoのPostgreSQLバージョンを17.0から18.1にアップグレードします。開発環境（Docker Compose）、CI/CD（GitHub Actions）、開発コンテナ（Dockerfile.dev）のすべてで使用するPostgreSQLバージョンを統一的に更新します。

**目的**:

- PostgreSQL 18の新機能やパフォーマンス改善を利用可能にする
- セキュリティパッチが最新の状態を維持する
- 依存ソフトウェアを最新バージョンに保つことで技術的負債を防ぐ

**背景**:

- 現在はPostgreSQL 17.0を使用している
- PostgreSQL 18.1がリリースされており、安定版として利用可能
- Go版とRails版の両方が同一のPostgreSQLインスタンスを共有しているため、統一的なアップグレードが必要

#### pg_dump `\restrict` 対策の更新

PostgreSQL 17.6以降で導入された `\restrict` / `\unrestrict` コマンドの対策について：

- **現状**: `go/Makefile` の `db-migrate` と `db-dump` ターゲットで、`sed` コマンドにより `\restrict` 行を削除している
- **PostgreSQL 18での状況**: PostgreSQL 18でも `pg_dump` は引き続き `\restrict` / `\unrestrict` コマンドを出力する（セキュリティ機能として定着）
- **dbmateの対応状況**: dbmate は `--restrict-key` オプションをサポート済み（PR #727がマージ済み）
- **対応方針**: 現在の `sed` による削除は引き続き機能するため、今回のアップグレードでは変更しない。dbmateの `--restrict-key` オプションへの移行は将来のタスクとする

| ファイル       | 変更内容                                                                           |
| -------------- | ---------------------------------------------------------------------------------- |
| `go/Makefile`  | コメントの `PostgreSQL 17.6` を `PostgreSQL 17.6以降` に更新（96-97行、123-124行） |
| `go/CLAUDE.md` | `\restrict` 対策の説明を `PostgreSQL 17.6` から `PostgreSQL 17.6以降` に更新       |

### PostgreSQL 18 の互換性確認

PostgreSQL 17から18へのアップグレードで注意すべき変更点：

- **データチェックサム**: PostgreSQL 18ではinitdbのデフォルトがデータチェックサム有効に変更。新規のDockerボリュームは自動的にこの設定で作成されるため、開発環境では問題なし
- **タイムゾーン略語**: セッションのタイムゾーン略語の優先順位が変更。Wikinoでは `TIMESTAMP WITH TIME ZONE` を使用しており、影響なし
- **文字の符号性（char signedness）**: クラスタが独自のデフォルトchar signedness設定を維持するようになった。新規のDockerボリューム作成のため、影響なし
- **全文検索**: ICUコレーションを使用している場合に動作変更の可能性。Wikinoではデフォルトのlibcコレーションを使用しており、影響なし

### Goドライバの互換性

- **lib/pq v1.10.9**: PostgreSQL 18をサポート
- **pgx/v5 v5.8.0**: PostgreSQL 18をサポート
- **River（riverdriver/riverpgxv5 v0.30.2）**: PostgreSQL固有のENUM型、JSONB、トリガーなどを使用しているが、PostgreSQL 18でも互換性あり

### Railsドライバの互換性

- **pg gem**: PostgreSQL 18をサポート

### データベースエクステンションの互換性

- **citext**: PostgreSQL 18で継続サポート
- **pgcrypto**: PostgreSQL 18で継続サポート
- **generate_ulid()関数**: 標準SQL関数のみ使用しており、互換性に問題なし

### 開発環境のデータ移行

- **新規ボリューム作成**: Dockerボリューム名を `postgresql17-data` から `postgresql18-data` に変更するため、既存のボリュームデータは使用されない
- **開発データの再作成**: 開発環境のデータは一時的なものであり、マイグレーションを再実行すればよい
- **テスト環境**: テスト用DBは `make test` 実行時に毎回リセットされるため、影響なし

### 実装しない機能（スコープ外）

<!--

以下の機能は今回の実装では**実装しません**：

- **dbmate `--restrict-key` オプションへの移行**: 現在の `sed` による `\restrict` 行削除で問題なく動作しているため、今回は対応しない。将来的にdbmateの `--restrict-key` オプションを利用して、より適切な方法でスキーマダンプの決定性を確保することを検討する

- **PostgreSQL 18 固有の新機能の活用**: アップグレード後に利用可能になる新機能の活用は、個別の設計書で対応する
