## 概要

<!--
ガイドライン:
- この機能が「何であるか」「なぜ必要か」を簡潔に説明
- 2-3段落程度で簡潔に

Go 版 Wikino では、フロントエンドのインタラクションに [Datastar](https://data-star.dev/)（v1.0.0-RC.7）を採用している。Datastar はサーバードリブンな hypermedia フレームワークであり、HTML の `data-*` 属性と Server-Sent Events（SSE）を使って、JavaScript を最小限に抑えた宣言的な UI を構築できる。

現在の Datastar の利用はフォーム送信時のボタン無効化（`$isSubmitting` シグナル）に限定されている。ページ編集画面のリンク一覧更新（タスク 8b-1）など、本来 Datastar の SSE + フラグメント更新パターンで実装すべき機能が、plain JavaScript の `fetch` + `innerHTML` で実装されてしまっている。これは Datastar の知見が不足していることが原因である。

この問題を解決するため、Datastar に特化した Claude Code スキルを作成する。スキルには Datastar の API リファレンス、Go バックエンドとの統合パターン、プロジェクト固有の実装ガイドラインを含め、Datastar を使った機能実装を正確に行えるようにする。

### スキルファイルの構成

スキルファイルは `.claude/commands/datastar.md` に配置する。

### スキルの内容構成

スキルファイルには以下のセクションを含める:

#### 1. スキルの概要と引数

- スキルの目的: Datastar を使った機能の実装・リファクタリングを支援する
- 引数: `$ARGUMENTS` で実装対象の機能説明を受け取る

#### 2. プロジェクト固有の情報

- Datastar バージョン: v1.0.0-RC.7
- ベンダーファイルの場所: `go/static/js/vendor/datastar-v1.0.0-RC.7.js`
- head.templ でグローバルに読み込み済み
- 既存の Datastar 使用パターン: フォーム送信時の `$isSubmitting` シグナル

#### 3. Datastar API リファレンス（主要な属性のみ）

Datastar の公式ドキュメントから、プロジェクトで使用する可能性が高い属性を厳選して記載する:

- **シグナル**: `data-signals`, `data-computed`
- **DOM 更新**: `data-text`, `data-show`, `data-class`, `data-attr`
- **イベント**: `data-on`（修飾子 `__debounce`, `__throttle`, `__passive` 含む）
- **SSE バックエンド連携**: `data-on:click="@get('/path')"` 等の `@get`, `@post`, `@patch`, `@delete`
- **フラグメント更新**: サーバーから返される SSE イベント形式（`datastar-merge-fragments`, `datastar-merge-signals`）

#### 4. Go バックエンドとの統合パターン

SSE エンドポイントの実装方法を具体的に記載する:

- **SSE レスポンスの書き方**: `Content-Type: text/event-stream` ヘッダー、`event:` と `data:` フィールドの形式
- **templ コンポーネントのフラグメント返却**: `templ.Component` を `bytes.Buffer` にレンダリングし、SSE フラグメントとして返す方法
- **CSRF 対策**: SSE リクエストでの CSRF トークンの扱い
- **既存のハンドラーパターンとの統合**: chi ルーターでの SSE エンドポイント登録

#### 5. 実装手順のチェックリスト

Datastar を使った機能を実装する際の標準的な手順:

1. HTML テンプレートに `data-*` 属性を追加
2. 必要に応じて SSE エンドポイントをハンドラーに追加
3. SSE レスポンスで返す templ フラグメントコンポーネントを作成
4. CSRF トークンの受け渡し確認
5. 動作確認

### Datastar Go SDK の導入

Datastar には公式の Go SDK（`github.com/starfederation/datastar/sdk/go`）が存在し、`MergeFragmentTempl()` 等のヘルパー関数を提供している。しかし、以下の理由から SDK の導入は見送る:

- **依存の最小化**: Go 版 Wikino は標準ライブラリを優先する方針であり、SSE レスポンスは標準の `net/http` で十分に実装可能
- **学習コスト**: SDK の抽象化を理解するよりも、SSE プロトコルを直接扱う方がシンプルで理解しやすい
- **バージョン追従**: SDK のバージョンと Datastar JS のバージョンの整合性を管理するコストが増える
- **YAGNI 原則**: 現時点で SDK が提供する機能の大半は不要

SDK の導入は、SSE エンドポイントが増えて共通化のメリットが明確になった段階で再検討する。

### スキルファイルの分割

API リファレンスを別ファイルに切り出し、スキルファイルからインクルードする方式も検討した。しかし Claude Code のスキルはシンプルな単一 Markdown ファイルが推奨されており、ファイル分割による複雑化を避けるため、1 ファイルに統合する。
