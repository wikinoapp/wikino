# docs ディレクトリガイド

このディレクトリでは「仕様書」「タスクリスト」「レビュー」の3種類のドキュメントを管理している。

## ディレクトリ構成

```
docs/
├── README.md         # このファイル
├── specs/            # 仕様書
│   ├── template.md   # 仕様書テンプレート
│   └── xxx.md        # 各機能の仕様書
├── tasks/            # タスクリスト
│   ├── template.md   # タスクリストテンプレート
│   └── xxx.md        # 各機能のタスクリスト
└── reviews/          # レビュー
    ├── template.md   # レビューテンプレート
    ├── done/         # レビュー完了済み
    └── xxx.md        # 各PRのレビュードキュメント
```

## 各ドキュメントの役割

| ドキュメント | 記述する内容 | ライフサイクル |
|---|---|---|
| 仕様書 (`specs/`) | 現在のシステムの状態と、なぜそうなっているか | 長期（継続的に更新） |
| タスクリスト (`tasks/`) | 何をどう変えるか（変更の差分） | 短期（タスク完了後にアーカイブ） |
| レビュー (`reviews/`) | 実装に対するセルフレビューの結果 | 短期（対応完了後に `done/` へ移動） |
| Git履歴 | 過去の状態 | 永続 |

仕様書とタスクリスト・レビューはライフサイクルが異なるため、独立したディレクトリで管理している。

## 仕様書 (`specs/`)

- 「現在のシステムの状態」と「なぜそうなっているか」を記述する
- 実装が完了したら、仕様書を最新の状態に更新する
- 過去の状態はGit履歴で参照する
- テンプレート: [specs/template.md](specs/template.md)

## タスクリスト (`tasks/`)

- 「何をどう変えるか」という変更内容を記述する
- 対応する仕様書へのリンクを「仕様書」セクションに記載する
- タスク完了後に対応する仕様書を新しい状態に更新する
- テンプレート: [tasks/template.md](tasks/template.md)

## レビュー (`reviews/`)

- 実装完了後のセルフレビュー結果を記録する
- ファイル名は `{ブランチ名}-{3桁連番}.md` の形式（例: `sign-up-1-1-001.md`）
- レビュー対応が完了したら `done/` ディレクトリに移動する
- テンプレート: [reviews/template.md](reviews/template.md)

## 運用フロー

1. 新しい機能を作る場合、仕様書とタスクリストを両方作成する
2. 既存機能を変更する場合、タスクリストを作成し対応する仕様書にリンクする
3. タスクを実装する
4. タスク完了後、仕様書を新しい状態に更新する
